1 System Context

This is a net-new platform for Payments Canada. It sits between financial institution participants as a central hub — not a bilateral integration. There is no existing Payments Canada R2P infrastructure to extend; every component is new.

The platform exposes REST APIs to participants and internally uses ISO 20022 message structures for all business events. It connects outward to a settlement rail (RTR or Lynx) for payment finality, and to a Proxy Directory for address resolution.

created→ sent→ delivered→ accepted→ payment_processing→ paid
delivered→ declined  |  deferred  |  expired  |  cancelled  |  payment_failed

2 New Components Required

R2P Request Service NEW

Core domain service. Accepts creation, modification, and cancellation calls. Validates against pain.013, enforces idempotency, generates UUID v7 transaction IDs, owns state machine transitions.

ISO 20022 Validator NEW

Stateless middleware. Validates pain.013 / pain.014 / camt.087 schemas. Returns structured error codes. Versioned independently — schema updates deploy without touching business logic.

Address Directory NEW

Resolves payer proxy (email / phone / alias) to participant endpoint and account reference. POC: in-memory static map seeded with 5 test participants. Production: live registry with TTL caching.

Routing Engine NEW

Delivers R2P messages to receiving participant via HTTPS. Maintains delivery state. Exponential backoff retry (3 attempts / max 30s). On exhaustion: marks failed, notifies originator.

Acknowledgement Service NEW

Receives POST /acknowledge from receiving participants. Records timestamped ack, transitions status sent → delivered, emits acknowledged event to Event Publisher.

Response Handler NEW

Receives accept / decline / defer from receiving participants. Validates expiry, amount, and state. Accept triggers Payment Execution Engine. Decline/defer notifies originating participant.

Payment Execution Engine NEW

Triggered exclusively by accepted R2P. Constructs real-time payment referencing R2P transaction ID. Submits to settlement rail (stubbed in POC). Updates status to paid or payment_failed.

Lifecycle State Manager NEW

Owns canonical state machine. Persists every transition with timestamp and actor. Runs expiry scheduler every 60s. Provides status query and history APIs.

Event Publisher NEW

POC: webhook delivery via HTTPS POST to participant-registered callback URLs. Production: Kafka per-participant with at-least-once delivery. Payload: ISO 20022-wrapped event envelope.

Audit Store NEW

Append-only log of every transition, API call, validation result, and system event. Never updated or deleted. POC: PostgreSQL with insert-only access. Production: immutable object storage + PG index.

4 Data Models

type R2PStatus =
  | 'created' | 'sent' | 'delivered'
  | 'accepted' | 'declined' | 'deferred'
  | 'expired' | 'cancelled'
  | 'payment_processing' | 'paid' | 'payment_failed'

interface R2PRequest {
  r2pId: string               // UUID v7
  idempotencyKey: string
  payerId: string
  payeeId: string
  originatingParticipantId: string
  receivingParticipantId: string
  amount: number
  currency: string            // ISO 4217
  dueDate: string             // ISO 8601 date
  expiryTimestamp: string     // ISO 8601 datetime
  remittanceInfo: string
  status: R2PStatus
  version: number             // optimistic locking
  createdAt: string
  updatedAt: string
}

interface R2PStateTransition {
  transitionId: string
  r2pId: string
  fromStatus: R2PStatus
  toStatus: R2PStatus
  actor: string               // participantId or 'system'
  timestamp: string
  reason?: string
}

interface R2PPayment {
  paymentId: string
  r2pId: string
  amount: number
  currency: string
  settlementRef: string
  status: 'processing' | 'settled' | 'failed'
  submittedAt: string
  settledAt?: string
}

interface EventSubscription {
  subscriptionId: string
  participantId: string
  callbackUrl: string
  eventTypes: string[]
  status: 'active' | 'suspended'
  createdAt: string
}

8 Risks & Mitigations

Risk 1 — State machine race conditions

Concurrent operations (e.g., payer responds while expiry job runs) can cause conflicting state transitions.

✓ Mitigation: Optimistic locking on R2PRequest.version column — reject the second write with 409 if version has changed since read.

Risk 2 — ISO 20022 schema versioning breaking participants

Schema updates may fail validation for participants that haven't yet upgraded their message format.

✓ Mitigation: Validator versioned as a separate deployable; support N and N-1 schemas simultaneously; 90-day deprecation window before removing old schema support.

Risk 3 — Webhook delivery failures silently dropping events

If a participant's callback URL is down, the Event Publisher could silently lose events.

✓ Mitigation: Persist every outbound event to OutboxEvent table before delivery; mark delivered only on HTTP 2xx; background sweeper retries undelivered events with exponential backoff up to 24 hours.

sequenceDiagram
  participant OP as Originating Participant
  participant REQ as R2P Request Service
  participant RTE as Routing Engine
  participant RP as Receiving Participant
  participant ACK as Acknowledgement Service
  participant RSP as Response Handler
  participant PAY as Payment Execution Engine
  participant SR as Settlement Rail
  participant EVT as Event Publisher

  OP->>REQ: POST /r2p/requests (pain.013)
  Note over REQ: status = created
  REQ->>RTE: route request to RP
  Note over REQ: status = sent
  RTE->>RP: deliver R2P request
  RP->>ACK: POST /r2p/requests/{id}/acknowledge
  Note over ACK: status = delivered
  ACK->>EVT: emit delivered event
  EVT-->>OP: webhook: delivered
  RP->>RSP: POST /r2p/requests/{id}/respond (accept)
  Note over RSP: status = accepted
  RSP->>PAY: trigger payment
  PAY->>SR: submit real-time payment
  SR-->>PAY: settlement confirmed
  Note over PAY: status = paid
  PAY->>EVT: emit paid event
  EVT-->>OP: webhook: paid
  EVT-->>RP: webhook: paid

End-to-end happy path: request created → routed → delivered → accepted → payment settled → both participants notified.

sequenceDiagram
  participant OP as Originating Participant
  participant REQ as R2P Request Service
  participant RTE as Routing Engine
  participant RP as Receiving Participant
  participant ACK as Acknowledgement Service
  participant RSP as Response Handler
  participant LCM as Lifecycle State Manager
  participant EVT as Event Publisher

  OP->>REQ: POST /r2p/requests (pain.013)
  Note over REQ: status = created
  REQ->>RTE: route request to RP
  Note over REQ: status = sent
  RTE->>RP: deliver R2P request
  RP->>ACK: POST /r2p/requests/{id}/acknowledge
  Note over ACK: status = delivered
  ACK->>EVT: emit delivered event
  EVT-->>OP: webhook: delivered

  alt Payer declines
    RP->>RSP: POST /r2p/requests/{id}/respond (decline)
    Note over RSP: status = declined
    RSP->>EVT: emit declined event
    EVT-->>OP: webhook: declined
  else Request expires before response
    LCM->>LCM: expiry scheduler detects expiryTimestamp < now
    Note over LCM: status = expired
    LCM->>EVT: emit expired event
    EVT-->>OP: webhook: expired
    EVT-->>RP: webhook: expired
    RP->>RSP: POST /r2p/requests/{id}/respond (attempt after expiry)
    RSP-->>RP: 409 EXPIRED
  end

Two failure paths: explicit payer decline and silent expiry with a subsequent late-response rejection.
