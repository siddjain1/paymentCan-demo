# Spec: 4.2 — POST /r2p/requests/{r2pId}/respond

**Ticket:** 4.2
**Type:** Backend API
**Points:** 5
**Depends on:** 4.1

---

## Endpoint

```
POST /r2p/requests/{r2pId}/respond
```

---

## Input

```typescript
{
  responseType: "accept" | "decline" | "defer"   // required
  participantId: string                            // required
  respondedAt:  string                             // required — ISO 8601 datetime
}
```

---

## Output

```typescript
{ r2pId: string, status: string }
```

---

## Errors

| Code | HTTP | Condition |
|------|------|-----------|
| VALIDATION_ERROR | 400 | responseType missing or not accept/decline/defer |
| NOT_FOUND | 404 | r2pId does not exist |
| EXPIRED | 409 | expiryTimestamp < now at time of response |
| INVALID_STATE_TRANSITION | 409 | status is not `delivered` |

---

## Logic

### Guards (evaluated in order)

1. **Fetch** — look up request by r2pId. Return 404 NOT_FOUND if missing.
2. **State guard** — status must be `delivered`. Anything else returns 409 INVALID_STATE_TRANSITION.
3. **Expiry check** — if `expiryTimestamp < now`, return 409 EXPIRED.
4. **Input validation** — responseType must be one of `accept`, `decline`, `defer`. Return 400 VALIDATION_ERROR if not.

### Persistence

Persist a `R2PResponse` row before any state mutation:

```typescript
{
  responseId:            string    // randomUUID()
  r2pId:                 string
  responseType:          "accept" | "decline" | "defer"
  respondingParticipantId: string
  respondedAt:           string
  createdAt:             string    // now
}
```

### State transitions

| responseType | from → to | status returned |
|---|---|---|
| `accept` | `delivered` → `accepted` | `"accepted"` |
| `decline` | `delivered` → `declined` | `"declined"` |
| `defer` | `delivered` → `deferred` | `"deferred"` |

### Post-transition actions

**accept:**
- Transition to `accepted`
- Trigger Payment Execution Engine (stub — ticket 5.1 not yet implemented; emit `PAYMENT_TRIGGERED` audit entry)
- Emit `RESPONSE_ACCEPTED` audit entry
- Emit `EVENT_RESPONDED_EMITTED` audit entry (originator notified, stub — ticket 6.5)

**decline:**
- Transition to `declined`
- Emit `RESPONSE_DECLINED` audit entry
- Emit `EVENT_RESPONDED_EMITTED` audit entry (originator notified, stub)

**defer:**
- Transition to `deferred`
- Emit `RESPONSE_DEFERRED` audit entry
- Emit `EVENT_RESPONDED_EMITTED` audit entry (originator notified, stub)

### Audit trail

All paths (success and failure) that mutate state must append:
- One `REQUEST_RESPONDED_{TYPE}` audit entry with `actor = participantId`
- One `EVENT_RESPONDED_EMITTED` audit entry with `actor = event-publisher`

State transition record appended with `from_status = "delivered"`, `to_status = {new status}`, `actor = participantId`.
