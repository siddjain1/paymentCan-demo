# Spec: 4.3 ‚ÄĒ Response Validation

**Ticket:** 4.3
**Type:** Backend API
**Points:** 3
**Depends on:** 4.2

---

## Overview

Extends `POST /r2p/requests/{r2pId}/respond` with an optional `amount` field and full
precondition validation. Guards 1 and 2 below already exist from ticket 4.2 and are
documented here for completeness. Guard 3 (amount) is new.

---

## Input change

```typescript
{
  responseType:  "accept" | "decline" | "defer"   // required (unchanged)
  participantId: string                             // required (unchanged)
  respondedAt:   string                             // required (unchanged)
  amount?:       number                             // NEW ‚ÄĒ optional
}
```

---

## Validation guards (evaluated in order)

| # | Condition | Error |
|---|-----------|-------|
| 1 | status ‚Č† `delivered` | 409 INVALID_STATE_TRANSITION *(existing ‚ÄĒ 4.2)* |
| 2 | `expiryTimestamp < now` | 409 EXPIRED *(existing ‚ÄĒ 4.2)* |
| 3a | `amount` present AND (`amount ‚Č§ 0` OR non-finite) | 400 VALIDATION_ERROR |
| 3b | `amount` present AND `amount ‚Č† request.amount` | 400 AMOUNT_MISMATCH |

Guard 3 runs **after** guards 1 and 2, **before** persistence.

---

## Amount validation rules (Guard 3)

- Field is fully optional ‚ÄĒ absent means no amount constraint applied.
- If present: must be a finite number greater than 0.
- If present and valid: must exactly equal `request.amount` (POC: no partial payments).
- Applies to all three `responseType` values.

---

## Error codes

| Code | HTTP | Condition |
|------|------|-----------|
| VALIDATION_ERROR | 400 | `amount` present but ‚Č§ 0 or non-finite |
| AMOUNT_MISMATCH | 400 | `amount` present but ‚Č† `request.amount` |
| INVALID_STATE_TRANSITION | 409 | status ‚Č† `delivered` *(existing)* |
| EXPIRED | 409 | `expiryTimestamp < now` *(existing)* |
| NOT_FOUND | 404 | r2pId unknown *(existing)* |

---

## Implementation notes

- `RespondToRequestInput` gains an optional `amount?: number` field.
- `RespondToRequestResult` error union gains `'AMOUNT_MISMATCH'`.
- Validation logic lives inside `respondToRequest()` ‚ÄĒ no new function needed.
- Route handler passes `amount` through only if present in request body.
