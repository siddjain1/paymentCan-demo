# Spec: 5.2 — Settlement Confirmation Handling

**Ticket:** 5.2
**Type:** Backend API
**Points:** 3
**Depends on:** 5.1

---

## Overview

No new endpoint. Extends `src/services/paymentEngine.ts` with a
`handleSettlementCallback(paymentId, success)` function that is called by the
settlement rail stub (and eventually a real RTR callback). The `submitPayment()`
fire-and-forget stub is updated to call this handler instead of emitting a bare
audit entry.

---

## New function

```typescript
export async function handleSettlementCallback(
  paymentId: string,
  success: boolean
): Promise<void>
```

---

## Logic

### 1. Lookup

- Fetch `R2PPayment` by `paymentId`. If not found → no-op (idempotent).
- Fetch r2p request by `r2pId`. If not found → no-op.
- If r2p status ≠ `payment_processing` → no-op (already handled or duplicate callback).

### 2. Success path (`success = true`)

| Action | Detail |
|--------|--------|
| Update R2PPayment | `status = "settled"`, `settled_at = now` |
| Transition r2p | `payment_processing` → `paid` |
| Audit | `SETTLEMENT_CONFIRMED` with `actor = "payment-engine"` |
| State transition record | `from: payment_processing`, `to: paid` |
| Notify both participants | `EVENT_PAID_EMITTED` audit entry, `actor = "event-publisher"` |

### 3. Failure path (`success = false`)

| Action | Detail |
|--------|--------|
| Update R2PPayment | `status = "failed"` |
| Transition r2p | `payment_processing` → `payment_failed` |
| Audit | `SETTLEMENT_FAILED` with `actor = "payment-engine"` |
| State transition record | `from: payment_processing`, `to: payment_failed` |
| Notify both participants | `EVENT_PAYMENT_FAILED_EMITTED` audit entry, `actor = "event-publisher"` |

---

## Changes to submitPayment()

Replace the current fire-and-forget stub block:

```typescript
// Before (5.1 stub):
void settlementRail(paymentId).then(({ success }) => {
  auditRepo.append({ event_type: 'SETTLEMENT_CALLBACK', ... })
})

// After (5.2):
void settlementRail(paymentId).then(({ success }) => {
  void handleSettlementCallback(paymentId, success)
})
```

---

## Idempotency

All guard checks are no-ops on missing records or wrong state — a duplicate
callback never double-transitions or double-notifies.

---

## Acceptance Criteria

- [ ] Settlement success transitions r2p status to `paid`
- [ ] Settlement failure transitions r2p status to `payment_failed`
- [ ] Both originating and receiving participants notified via Event Publisher (stubbed)
- [ ] All transitions recorded in R2PStateTransition and AuditStore
- [ ] Duplicate/late callbacks are no-ops (idempotent)
- [ ] All unit tests pass
